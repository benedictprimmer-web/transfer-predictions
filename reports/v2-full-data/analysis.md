# V2 Full-Data Analysis — Synthesis

Generated by `python3 -m validate.v2_full_data` on the combined
`agent/data-audit-pr` branch. Descriptive only — no model fit, no
locked-period metric.

**CORRECTED 2026-07-13** (external review, reproduced independently): the
original version of this document (a) combined low-minutes and
unknown-minutes rows into one "~40%" figure and (b) called
`transfer_performance_link_safe` a blocked destination-season source, which
was a mischaracterization (see `docs/contradiction-log.md`). Both fixed
below. `docs/v3-plan.md`/`docs/v3-results.md` build the actual
destination-season table this document originally assumed was blocked.

## 1. Coverage

FACT — `fbref_perf` covers 1,848 (season, league, role, minutes-band) cells
across seasons 2010-2026 and 5 `Comp` values. Source:
`reports/v2-full-data/coverage_cube.csv`.

FACT (corrected — separated, not combined) — of 52,951 `fbref_perf` rows,
**14,416 (27%) are LOW-minutes** (`<450`, minutes value known and small)
and a **separate 6,699 (12.6%) have UNKNOWN minutes** (the field is null —
`reports/v2-full-data/missingness_patterns.csv`: `minutes_known` =
46,257/52,951 = 87.36%). These are different problems requiring different
treatment (task's own distinction, correctly implemented as of this
correction): low-known-exposure gets partial pooling
(`impact.evidence.GammaPoissonShrinkage`); unknown exposure gets
abstention (`support_flag() == "unknown_exposure"`), never a fabricated
per-90 rate. The original analysis's "~40% low-or-unknown" framing invited
exactly the conflation the task warned against, and a real bug followed
from it: `impact/evidence.py::support_flag` previously let `NaN` minutes
fall through to `"in_distribution"` (the *highest*-confidence label,
because `NaN < threshold` evaluates `False` in Python/numpy) — fixed.

INFERENCE — a hard 450-minute cutoff alone would discard 27%+ of rows
outright; exposure-aware shrinkage is the better treatment for the
known-low-minutes population specifically. It does nothing for the
unknown-minutes population, which needs source repair or explicit
abstention, not a statistical fix.

## 2. Percentile cohort viability

FACT (corrected — tiered, not a single cutoff) — of 439 (season, league,
role) cells: 265 `high` confidence (100+ players), 88 `moderate` (50-99),
52 `low` (20-49), 34 `unavailable` (<20). Source:
`reports/v2-full-data/cohort_support.csv`. The original "405/439 (92%)
viable at a 20-player minimum" framing treated a bare 20-player cohort as
equivalent to a 300-player one; `impact.evidence.domestic_percentile` and
`cohort_support()` now both apply the task's four-tier scheme.

INFERENCE — domestic percentiles are broadly *available* within the
5-league `fbref_perf` footprint, but "available" spans a wide confidence
range; a `low`-tier percentile (20-49 players) should be displayed with
visibly less certainty than a `high`-tier one. This footprint is also
*small* relative to the 134,147-row canonical transfer table
(`docs/data-audit.md`) — most canonical transfers involve at least one side
outside these 5 competitions.

## 3. Metric distributions — raw vs per-90

FACT — for the five per-90-eligible count metrics tested (non-penalty xG,
xA, progressive actions, tackles, interceptions),
`reports/v2-full-data/metric_distributions.csv` now separately reports
`n_unknown_minutes` (rows where the metric is known but minutes is null —
these were previously silently dropped from the per-90 distribution with
no count of how many) and `n_low_minutes_under_450` (known-low-exposure
rows, included in the per-90 distribution but noted as unstable).

INFERENCE — none of these five metrics is exempt from the low-vs-unknown
distinction in §1; the same split affects every candidate evidence-card
dimension, not just one.

## 4. Cross-league mover support

FACT — 48 origin-destination league pairs have 5+ movers in
`transfers_canonical` (all transfer types, all seasons); those 48 pairs
cover 6,134 movers total, of which 1,146 fall in the locked period
(`season >= 2023`, correctly excluded from any fitting — see
`locked_test_audit.json`). Source: `reports/v2-full-data/mover_support.csv`.

FACT — **data-quality finding**: `transfers_canonical.from_league`/`to_league`
contains two distinct string values for the same competition —
`"La Liga"` (used 1994-2022) and `"LaLiga"` (used exclusively from
season 2023 onward, matching the competition's real-world rebrand). This
silently splits one league into two cohorts for any league-keyed grouping
that spans the rebrand year, and would also fragment `transfer_performance_
outcomes_future`'s destination-club matching (`docs/v3-plan.md`) for any
La Liga destination transfer dated 2023+.

INFERENCE — genuine defect in `ingest/merge.py`'s canonical normalization,
not a modelling choice. Not fixed in this PR (would re-churn counts
already reconciled in `docs/reconciliation.md`). Logged in
`docs/contradiction-log.md`.

## 5. Locked-period protection

FACT — `reports/v2-full-data/locked_test_audit.json`: of 134,147 canonical
transfer rows, 111,274 are development rows (`season < 2022`), 3,759 are
the deliberate `season == 2022` buffer, and 19,114 are the locked period
(`season >= 2023`), excluded by row-key hash only — no locked-row outcome
was read or computed.

FACT (corrected) — `fbref_perf` (2010-2026) **does overlap** the locked
period; the original document did not flag this because it wrongly assumed
`fbref_perf`'s window was 2017-22 (see `docs/contradiction-log.md`
provenance finding). Any code that reads `fbref_perf` for a dev-only
computation must route through `validate.locked_guard`, matched by season;
`validate/v3_sporting_target.py` does this explicitly rather than relying
on a (previously wrong) coverage-window assumption.

## 6. What full-data analysis could not do here — corrected

The original version of this section said `transfer_performance_link_safe`
being unavailable blocked "any destination-season sporting/availability
target funnel." That's not accurate: this specific table was never a
destination-season source (§0 of `docs/mustermann.md`, corrected). A real
destination-season outcome table is buildable from in-repo data — see
`docs/v3-plan.md` for the construction and `docs/v3-results.md` for the
resulting effective sample and whether it supports a sporting prototype.
